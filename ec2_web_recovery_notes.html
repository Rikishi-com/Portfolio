EC2サーバの復旧・Webサーバ再構築作業メモ

概要

旧EC2インスタンスに障害が発生し、SSHおよびWeb接続が不能となったため、以下の手順で新インスタンスにて復旧作業を実施した。

新インスタンスの作成とElastic IPの再アタッチ

  • 新しいAmazon Linux 2023インスタンスを起動
  • Elastic IP（57.180.208.0）を再アタッチ

古いインスタンスのボリュームを新インスタンスにアタッチ

アタッチ時のデバイス名 

/dev/xvdf

/dev-デバイスファイルが格納されているディレクトリ(Linux標準)

  • LinuxではUSBやネットワーク機器，HDDなどの物理的なデバイスもファイルとして扱う
  • それらはすべて/devというフォルダに存在する
種類 コマンド
一般的なHDD sda
EC2のOSが入っているディスク xvda
自身で追加したEBSボリューム xvdf

xvd-EC2で使われる仮想ディスクのプレフィックス(xen virtual disk)
f-接続された順番による識別子(a,b,c...,f)→fは6台目のディスク

  • aは1番目なのでOS用に割り振られている

ディスクの確認 

lsblk

lsblk-デバイスの状態を確認するコマンド

マウント処理 

sudo mkdir /mnt/rescue
sudo mount /dev/xvdf1 /mnt/rescue

sudo mkdir /mnt/rescue-マウントするディレクトリを作成する
mount-/dev/xvdf1ではブロックデバイスがあるかどうか見るだけなので，mountを使用することで中のファイルやフォルダを見ることができる

  • イメージとしては，外付けHDDxvdfがdev/においてあるだけ．それをPCに差し込む処理がmount

データ確認と抽出

sshd設定の確認 

sudo cat /mnt/rescue/etc/ssh/sshd_config

cat-ファイルの中身を表示するコマンド．編集ではない

→ 設定ファイルに問題ない場合，sshdサービスが停止していた可能性が高い．

Webコンテンツの確認 

ls /mnt/rescue/var/www/html

新環境へWebファイルを移行

/var/www/html が存在しない場合は作成 

sudo mkdir -p /var/www/html

-p-一気にすべてのフォルダを作成できる

ファイルのコピー 

sudo cp -r /mnt/rescue/var/www/html/* /var/www/html/

パーミッション設定

Amazon Linux 2023 環境のため ec2-user を所有者に設定：

sudo chown -R ec2-user:ec2-user /var/www/html
sudo chmod -R 755 /var/www/html

chown-change owner　=　所有者を変更するコマンド
ec2-user:ec2-user-ユーザー名:グループ名
chmod-change mode　=　パーミッション(アクセス権)を変更するコマンド
755-アクセス権の設定

数字 意味 説明
7 rwx 読む，書く，実行
5 rx 読む，実行

Apacheの起動と確認

Apacheの起動 

sudo systemctl start httpd

自動起動の有効化 

sudo systemctl enable httpd

状態確認 

sudo systemctl status httpd

トラブルシュート

ssh のホストキー警告対応 

ssh-keygen -R [Elastic IP]

chown: invalid user エラー

→ apache:apache は Amazon Linux 2023 では存在しない。代わりに ec2-user を使用。

cp: target is not a directory エラー

→ /var/www/html/ が存在していなかったため、mkdir -p で作成。

確認方法

ローカルからWebページ確認 

http://[Elastic IP]/

補足

  • 今回の復旧では /mnt/rescue/ 配下の旧環境を活用し、データを安全に抽出できた。
  • サーバを切り替えた際には known_hosts のホストキーエラーに注意する。
  • sshd 停止やファイアウォールの設定誤りが原因で接続不能になることがある。
